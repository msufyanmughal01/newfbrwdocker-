## Custom Application Extensions

These are additional columns/tables beyond what Better Auth provides, defined in application Drizzle schema.

### organization (extended fields — stored in `metadata` JSON or separate table)

For Phase 1, organization-specific business fields (tax identifier, contact info) will be stored in a separate `organizationProfile` table to keep Better Auth's organization table clean.

#### organizationProfile

| Column         | Type              | Constraints                    | Notes                    |
|----------------|-------------------|--------------------------------|--------------------------|
| id             | text              | PK                             |                          |
| organizationId | text              | NOT NULL, FK → organization.id, UNIQUE | 1:1 with org    |
| taxIdentifier  | varchar(50)       | NULLABLE                       | NTN/STRN number          |
| phone          | varchar(20)       | NULLABLE                       | Contact phone            |
| address        | text              | NULLABLE                       | Business address         |
| city           | varchar(100)      | NULLABLE                       |                          |
| status         | text              | NOT NULL, DEFAULT "active"     | "active", "suspended"    |
| createdAt      | timestamp         | NOT NULL, DEFAULT now()        |                          |
| updatedAt      | timestamp         | NOT NULL                       |                          |

## Validation Rules

- **email**: Must be valid email format (RFC 5322 simplified)
- **password**: Minimum 8 characters (enforced by Better Auth config)
- **role**: Must be one of: "owner", "operator", "accountant"
- **taxIdentifier**: Optional in Phase 1; format validation deferred
- **invitation.status**: Transitions: pending → accepted | canceled | expired (no reverse transitions)
- **organizationProfile.status**: "active" or "suspended" only

## Tenant Scoping Rules

- All queries for business data MUST include `WHERE organizationId = :currentOrgId`
- The `member` table establishes user ↔ organization relationship
- Active organization is determined from the user's single membership (Phase 1: one org per user)
- Session → User → Member → Organization chain resolves the current org context

## Financial Column Standards

For future phases (invoices, payments), all monetary columns MUST use:
- Drizzle type: `decimal("column_name", { precision: 18, scale: 2 })`
- Storage: String representation to avoid floating point
- Computation: Server-side only (Constitution Principle IX)
